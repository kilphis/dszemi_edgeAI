## 1. プロジェクト概要

### テーマ

Sony AITRIOS エッジAIカメラを使い、熊本市の駐輪場において  
**違法駐輪（禁止エリアへの侵入）** と **放置自転車（長時間の停留）** をリアルタイムで自動検出するシステムを構築する。

### 対象自治体の背景

- 熊本市は全国的にも渋滞が多く、自転車利用を積極的に推進している
- 駐輪場のスペースが慢性的に不足。主原因は駐輪場内に長期間放置された自転車
- 放置自転車対策費の約80%が「巡回指導員の人件費」に費やされている
- その結果、駐輪場新設・老朽設備修繕への予算が削られ、スペースがさらに減る悪循環
- アーケード周辺などでは駐輪禁止エリアへの無断駐輪が常態化しているが、指導員の巡回タイミングとずれると見逃される

### 提案するソリューション

エッジAIカメラで24時間自動監視し、次の3機能を実装する。

| 機能 | 説明 |
|---|---|
| 違法駐輪検出 | 駐輪禁止エリアに自転車が入った瞬間をリアルタイムで検出・記録 |
| 放置自転車検出 | 同じ座標に長時間とどまっている自転車を追跡し、閾値超過でアラート |
| ヒートマップ | どこに自転車が集中しているかを可視化し、設備改善の根拠を提供 |

**既存のセンサー式装置との差別化点：** カメラ1台でどんな形の簡易駐輪場にも低コストで導入可能。座標設定を変えるだけで別の場所にも再利用できる。

## 2. 使用素材・要件定義

### カメラ

- **Sony AITRIOS** エッジAIカメラ
- 推論結果はカメラ内で計算され、クラウド（AITRIOSコンソール）経由でJSONとして取得
- 画像解像度：320×320ピクセル（0〜320の座標系）

### 学習クラス

- **クラス0：bicycle（自転車）** のみ
- 向き・個体識別は行わない（存在する／しない の2値検出）

### 背景画像

- 駐輪可能範囲と駐輪禁止範囲が明確に分かれている駐輪場を選定
- カメラは**斜め上方**から撮影（真俯瞰ではない）

### 駐輪禁止エリアの設計思想

モデルには禁止エリアを学習させず、**事前に座標として与える設計**にした。  
これにより：
- 場所が変わっても座標だけ更新すれば再利用できる
- モデルの再学習が不要

## 3. システム構成

```
AITRIOSカメラ（エッジ推論）
       ↓  REST API（Console API v2）
download_inferences.py
       ↓  JSON + 画像を保存
data/<device_id>/<timestamp>/
    images/                  ← カメラ画像（JPEG）
    inferences/deserialized_inferences/  ← 推論結果JSON
       ↓
bike_parking.py / demo.py / demo_abandoned.py
       ↓
output/
    bike_heatmap.png    ← ヒートマップ
    bike_count.png      ← フレームごとの検出数グラフ
    bike_frame_NNN.png  ← バウンディングボックス重畳画像
```

### 推論JSONの形式

```json
{
  "T": "2026-06-16T01:37:30.349+09:00",
  "1": {"C": 0, "P": 0.609, "X": 7, "Y": 232, "x": 88, "y": 310},
  "2": {"C": 0, "P": 0.521, "X": 120, "Y": 180, "x": 200, "y": 260}
}
```

- `T`：タイムスタンプ
- `C`：クラスID（0=bicycle）
- `P`：確信度（0〜1）
- `X, Y`：バウンディングボックス左上の座標
- `x, y`：バウンディングボックス右下の座標

### 技術的工夫①：ポリゴンによるゾーン定義

#### なぜ四角形（矩形）を使わないのか

カメラが斜め撮影のため、現実の「まっすぐな通路」が画像上では**台形**に見える。  
矩形ゾーンでは本来合法なエリアまで禁止ゾーンに含まれてしまう（誤検知が増える）。

```
現実の通路（真上から）   →   カメラ画像（斜めから）
 ┌──────────────────┐         ／￣￣￣￣￣￣￣＼
 │                  │   →   ／                ＼
 └──────────────────┘        ＼_______________／
       長方形                       台形
```

#### 解決策：ポリゴン定義

頂点の座標をリストで並べることで、台形・五角形など任意の形を定義できる。

```python
# 矩形（斜め映像に弱い）
{"name": "aisle", "X": 0, "Y": 240, "x": 320, "y": 320}

# ポリゴン（台形にできる）
{"name": "zone_2", "polygon": [(80, 319), (220, 212), (171, 190),
                                (181, 186), (228, 204), (253, 194),
                                (274, 203), (266, 319)]}
```

#### 点がゾーン内にあるかの判定：レイキャスティング法

自転車の重心座標 `(cx, cy)` が禁止ゾーン内かどうかを判定する。

**アイデア：** 点から右向きに光線（レイ）を飛ばす。  
光線がポリゴンの辺と**奇数回**交われば内側、**偶数回**なら外側。

```python
def point_in_polygon(px, py, polygon):
    inside = False
    n = len(polygon)
    j = n - 1
    for i in range(n):
        xi, yi = polygon[i]
        xj, yj = polygon[j]
        if ((yi > py) != (yj > py)) and \
           (px < (xj - xi) * (py - yi) / (yj - yi) + xi):
            inside = not inside
        j = i
    return inside
```

| | 矩形ゾーン | ポリゴンゾーン |
|---|---|---|
| 定義の方法 | 左上・右下の2点 | 頂点のリスト（何点でも） |
| 対応できる形 | 長方形のみ | 台形・五角形・任意の多角形 |
| 斜めカメラへの対応 | ✗ ズレが生じる | ✓ 形に合わせられる |
| 判定アルゴリズム | 矩形の重なり計算（IoU） | レイキャスティング法 |

### 技術的工夫②：座標近傍マッチングによる個体追跡

AITRIOSカメラの推論結果はフレームごとの座標のみ。**同一個体のIDは持たない。**

そこで「前フレームと今フレームで30px以内に検出されたら同一個体」とみなす  
**座標近傍マッチング（Coordinate Proximity Matching）** を実装した。

```
フレーム1: cx=120, cy=200 に自転車 → track_id=0 として登録
フレーム2: cx=118, cy=203 に自転車 → 30px以内 → track_id=0 の滞在時間を更新
フレーム3: cx=122, cy=201 に自転車 → 30px以内 → track_id=0 の滞在時間を更新
```

これにより「この自転車は何分間ここにいるか」を計算できる。

---

### 技術的工夫③：ヒートマップの仕組み

全フレームの全バウンディングボックスを320×320のグリッドに累積する。

```
各フレーム:  BB内のすべてのピクセルに +1 する
全フレーム合計後: 数値が大きい = 自転車がよく存在した場所
```

累積グリッドをカラーマップ（Blues）で描画することで、  
**どこに自転車が集中しているか** を一目で把握できるヒートマップを生成する。

## 5. 現在の課題と改善アイデア

### 課題①：最近傍が保証されない

現在の実装は「30px以内の最初のトラック」にマッチする。  
複数の自転車が近くにいる場合、距離が一番近いトラックではなくリストの先頭に当たることがある。

**改善策：** 全検出 × 全トラックの距離を計算し、コスト最小の組み合わせで割り当てるハンガリアン法を使う。

### 課題②：消えた自転車のトラックが残り続ける

一度登録されたトラックは永遠に消えないため、去った自転車と後から来た別の自転車が「同一個体」と誤認識される可能性がある。

**改善策：** `last_seen` から一定フレーム以上検出されなければトラックを消滅させる。

### 課題③：密集時の誤マッチ

自転車が隣り合って止まっていると、30px の半径が重なり、誤マッチが起きる。

**改善策：** マッチング半径をバウンディングボックスのサイズに応じて動的に変えるか、IoUでマッチングする。

| # | 課題 | 影響 | 改善アイデア |
|---|---|---|---|
| 1 | 最近傍が保証されない | 誤トラック割り当て | ハンガリアン法 |
| 2 | トラックが消えない | 別個体を継続と誤認識 | 一定フレーム未検出でトラック消滅 |
| 3 | 密集時の誤マッチ | 滞在時間の誤算 | IoUベースのマッチング |

# 個体識別の難しさと追跡アルゴリズムの比較

> 技術者向け補足資料

---

## なぜ個体識別が難しいのか

Sony AITRIOS カメラの推論出力は「**このフレームにこの座標で自転車がある**」という情報だけで、
**フレームをまたいだ個体のID を持たない**。

```
フレーム1:  (cx=120, cy=200) に自転車
フレーム2:  (cx=118, cy=203) に自転車  ← 同じ自転車？ 別の自転車？ わからない
```

「放置自転車」を検出するには「同じ自転車が N 分間ここにいた」という時系列情報が必要なので、
フレームをまたいで個体を同定するロジック（**トラッキング**）を自前で実装する必要がある。

---

## 選択肢として存在するアルゴリズム

### ① Naive 最近傍法（旧実装）

**やること:** 各検出について、トラックリストを先頭から走査し、`COORD_MATCH_RADIUS`（30px）以内の**最初の**トラックに割り当てる。

```
検出: cx=105
トラックA: cx=100  (距離5px)  ← リスト先頭 → こちらにマッチ
トラックB: cx=102  (距離3px)  ← より近いが無視される
```

| 項目 | 内容 |
|---|---|
| 計算量 | O(n × m)  n=検出数, m=トラック数 |
| 保証 | なし（リスト順に依存） |
| 欠点 | リストの先頭にあるトラックが有利になる |

---

### ② Greedy 最近傍法（現在の実装）

**やること:** 全検出 × 全トラックの距離を計算し、近い順にソートしてからグリーディーに割り当てる。

```
全ペアの距離を列挙:
  (検出A, トラック1): 3px  ← 最小 → 割り当て
  (検出A, トラック2): 8px
  (検出B, トラック2): 5px  ← 次の最小 → 割り当て
  ...
```

| 項目 | 内容 |
|---|---|
| 計算量 | O(n × m × log(n×m)) |
| 保証 | 「最も近いペアから優先」は保証される |
| 欠点 | **大域的な最適解は保証されない**（局所的に近いペアを選んだ結果、全体のコストが増える場合がある） |

---

### ③ ハンガリアン法（未実装）

**やること:** 検出とトラックの距離行列を作り、**総距離の合計が最小になる組み合わせ**を求める。

```
距離行列:
         トラック1  トラック2
検出A  [   3px        8px   ]
検出B  [   7px        5px   ]

greedy の選択: A→1(3), B→2(5)  合計=8
Hungarian:    A→1(3), B→2(5)  合計=8  (この例では同じ)

複雑な場合:
検出A  [   3px        4px   ]
検出B  [   2px       10px   ]

greedy: A→1(3) → B→2(10)  合計=13  ← 局所最適で失敗
Hungarian: A→2(4), B→1(2)  合計=6  ← 大域最適
```

| 項目 | 内容 |
|---|---|
| 計算量 | O(n³) |
| 保証 | **大域的な最適解（総コスト最小）が保証される** |
| 欠点 | 実装コストが高い。n, m が小さければ greedy と差が出にくい |
| 必要な場面 | 3台以上が密集していて、どのペアを選ぶかで総コストが変わる状況 |

---

### ④ IoU ベースのマッチング（未実装）

**やること:** 重心間の距離の代わりに、バウンディングボックス同士の**重なり率（IoU）** でマッチングする。

```
IoU = (BBの重なり面積) / (BBの合計面積 - 重なり面積)

IoU が高い = 同じ位置・同じサイズの BB = 同一個体の可能性が高い
```

| 項目 | 内容 |
|---|---|
| 保証 | 距離より BB サイズの変化に頑健 |
| 欠点 | BB サイズが安定していないと機能しない（カメラ角度・遮蔽で変動する） |
| 必要な場面 | 密集した駐輪場で BB が隣接しているが重なりは少ない場合 |

## 今回のデータでの比較結果

`eval_tracker.py` で同一データ（50フレーム・約2.7分・自転車196件）を比較した結果：

```
strategy    dead_after  total_tracks  short(<5s)  abandoned  avg_stay_sec
----------  ----------  ------------  ----------  ---------  ------------
naive       0           23            7           11         56.8    ← 旧実装
naive       5           40            20          10         17.2
greedy      0           22            4           14         67.9    ← 現在
greedy      5           36            14          9          19.4
```

**指標の読み方:**

| 指標 | 意味 | 良い方向 |
|---|---|---|
| `total_tracks` | 全フレームで登録されたトラック総数 | 少ない（誤マッチが少ない） |
| `short(<5s)` | 5秒未満で消えたトラック数 | 少ない（誤マッチによる幽霊トラックが少ない） |
| `abandoned` | 放置アラートの発火回数 | 実態に近い数 |
| `avg_stay_sec` | 全トラックの平均滞在時間 | 長い（同一個体を安定追跡できている） |

**読み取れること:**

- `naive → greedy` で `short(<5s)` が **7→4** に減少：誤マッチによる短命トラックが減った
- `avg_stay_sec` が **56.8→67.9** に増加：同じ個体をより長く追跡できている
- `dead_after=5` は今回のデータ（50フレーム）では逆効果：データが短すぎてトラックが死ぬたびに新規登録され、`total_tracks` が逆に増える。データが長くなれば有効になる

---

## まとめ：アルゴリズムの選択指針

```
自転車の密度が低い（1〜3台）
  → greedy で十分

自転車が密集（4台以上が近接）
  → Hungarian を検討

カメラが近く BB が安定している
  → IoU ベースを検討

データが長時間にわたる（数時間以上）
  → dead_after を有効にして幽霊トラックを消す
```

今回のケース（50フレーム・密度は低め）では **greedy + dead_after=0** が最もバランスが良い。

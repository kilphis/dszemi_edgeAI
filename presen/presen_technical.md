```

    ▼

    │ GET /images          → JPEG画像
    │ GET /inferences      → Base64エンコードされたFlatBuffers
    ▼
[download_inferences.py]
    │ FlatBuffers → JSON にデシリアライズ
    │ 認証トークンをキャッシュ（3500秒でリフレッシュ）
    ▼
data/<device_id>/<timestamp>/
    ├── images/                         ← JPEG画像
    └── inferences/deserialized_inferences/  ← 推論JSON
    ▼
[bike_parking.py / demo.py / demo_abandoned.py]
    ▼
output/
    ├── bike_heatmap.png
    ├── bike_count.png
    └── bike_frame_NNN.png
```

### 推論JSONの構造

```json
{
  "T": "2026-06-16T01:37:30.349+09:00",
  "1": {"C": 0, "P": 0.609, "X": 7,  "Y": 232, "x": 88,  "y": 310},
  "2": {"C": 0, "P": 0.521, "X": 120, "Y": 180, "x": 200, "y": 260}
}
```

| キー | 意味 |
|---|---|
| `T` | タイムスタンプ（ISO 8601） |
| `C` | クラスID（0 = bicycle） |
| `P` | 確信度（0〜1） |
| `X, Y` | バウンディングボックス 左上 |
| `x, y` | バウンディングボックス 右下 |

**座標系**: 0〜320 の正規化座標。画像解像度によらず同じスケール。

## Tech-2: 禁止ゾーン判定の設計判断

### なぜ矩形（IoU）ではなくポリゴン + Point-in-Polygon か

カメラが斜め撮影のため、現実の長方形エリアが画像上では台形になる。
矩形ゾーンでは誤検知が増えるため、任意の多角形で定義できる設計にした。

```
現実（真上から）         カメラ画像（斜めから）
 ┌──────────────┐            ／￣￣￣￣￣＼
 │  禁止エリア  │   →       ／  禁止エリア ＼
 └──────────────┘           ＼_______________／
      長方形                      台形
```

ゾーン定義の変化:

```python
# 矩形（旧）
{"name": "zone", "X": 0, "Y": 240, "x": 320, "y": 320}

# ポリゴン（現在）
{"name": "zone_2", "polygon": [
    (80, 319), (220, 212), (171, 190),
    (181, 186), (228, 204), (253, 194),
    (274, 203), (266, 319)
]}
```

### レイキャスティング法による点判定

自転車の重心 `(cx, cy)` がポリゴン内かどうかを判定する。

**アルゴリズム:** 点から右向きにレイを飛ばし、辺との交差が奇数回なら内側。

```python
def point_in_polygon(px, py, polygon):
    inside = False
    n = len(polygon)
    j = n - 1
    for i in range(n):
        xi, yi = polygon[i]
        xj, yj = polygon[j]
        if ((yi > py) != (yj > py)) and \
           (px < (xj - xi) * (py - yi) / (yj - yi) + xi):
            inside = not inside
        j = i
    return inside
```

17行、O(n)、外部ライブラリ不要。n はゾーンの頂点数（今回は最大8頂点）。

### ゾーン座標のキャリブレーション手順

`pick_zone.py` を実行 → 実際のカメラ画像をクリックして頂点を選択 → コードに貼り付け

```bash
uv run --no-project scripts/pick_zone.py
```

## Tech-3: 個体追跡の設計と課題

### 座標近傍マッチング（Coordinate Proximity Matching）

AITRIOSの推論出力にはフレーム間の個体IDが存在しない。
そこで「前フレームから30px以内に検出されたら同一個体」とみなすロジックを自作した。

```
フレームN:   cx=120, cy=200  → traあるg
ck_id=0 として登録
フレームN+1: cx=118, cy=203  → 距離 ≈ 2.8px < 30px → track_id=0 の滞在時間を更新
フレームN+2: cx=122, cy=201  → 距離 ≈ 2.2px < 30px → track_id=0 の滞在時間を更新
```

閾値 `COORD_MATCH_RADIUS = 30px`（0〜320スケール）は実データから経験的に設定。

---

### 現在の実装の課題

#### 課題1: 最近傍が保証されない

現在: リストを先頭から走査し、30px以内の**最初の**トラックにマッチ

```
トラックA: cx=100  (距離12px)  ← リストが先頭 → こちらにマッチ（誤り）
トラックB: cx=108  (距離 4px)  ← 本来こちらが正解
```

改善案: **ハンガリアン法** で全検出×全トラックの距離行列を最小コストで割り当て

#### 課題2: 消えたトラックが残り続ける

現在: 一度登録されたトラックは永続する

```
09:00  自転車Aが来る   → track_id=0 登録
09:30  自転車Aが去る   → track_id=0 はリストに残ったまま
10:00  自転車Bが来る   → track_id=0 近くなら「A継続」と誤認識（本当はBが新規）
```

改善案: `last_seen` から N フレーム未検出でトラックを **Dead** 状態に遷移

#### 課題3: 密集時の誤マッチ

30px の半径が隣接自転車と重なると、どちらのトラックにマッチするか不定になる。

改善案: IoUベースのマッチング、またはBBサイズに応じた動的半径

---

### 課題の整理と現在の対応状況

| # | 課題 | 現在の影響 | 対応状況 |
|---|---|---|---|
| 1 | 最近傍が保証されない | 誤トラック割り当て | ✅ **greedy 戦略で改善**（全ペア比較→近い順割り当て）。完全な最適解はハンガリアン法（将来課題、Tech-5 参照） |
| 2 | トラックが消えない | 別個体を同一と誤認識 | ✅ **`dead_after` パラメータで実装済み**。短いデータでは再登録が増えるため現在は `dead_after=0` 推奨 |
| 3 | 密集時の誤マッチ | 滞在時間の誤算 | ❌ **未対応**。IoUベースのマッチングが有効（将来課題、Tech-6 参照） |

**greedy vs naive の定量比較（eval_tracker.py、50フレーム・threshold=30s）:**

| strategy | 総トラック数 | 短命トラック(<5s) | 放置検出数 | 平均滞在 |
|---|---|---|---|---|
| naive / dead_after=0 | 23 | 7 | 11 | 56.8s |
| greedy / dead_after=0 | 22 | **4** | **14** | **67.9s** |

greedy により誤マッチ由来の短命トラックが減り、滞在時間計算の精度が向上した。

## Tech-4: 拡張性・再利用性

### 場所の変更 → 座標だけ更新すれば再利用できる

```python
# bike_parking.py の設定部分を変えるだけ
NO_PARKING_ZONES = [
    {"name": "zone_A", "polygon": [(x1,y1), (x2,y2), ...]},
    {"name": "zone_B", "polygon": [(x1,y1), (x2,y2), ...]},
]
```

モデルの再学習・デプロイは不要。`pick_zone.py` で新しい座標を取得して貼り替えるだけ。

### 閾値の調整

```python
STAY_TIME_THRESHOLD_MINUTES = 60   # 放置判定の閾値（分）
COORD_MATCH_RADIUS = 30            # 同一個体と判定する距離（px, 0〜320スケール）
```

これらは設定値として分離されており、場所・用途に応じて変更しやすい。

### 複数カメラへの拡張

**現在できること**: データ取得の並列化のみ。

```
data/
  <device_id_A>/   ← カメラA（入口付近）のデータ
  <device_id_B>/   ← カメラB（奥のエリア）のデータ
```

`download_inferences.py` に複数のデバイスIDを渡すことで、カメラごとの推論データを取得できる。ただし**追跡はカメラごとに独立して動く**。

**将来課題: カメラをまたいだ同一個体の追跡**

カメラAの視野からカメラBの視野に移動した自転車は、現在の実装では別の個体として新規登録される。これを解決するには、カメラ間で共通のIDを付与する仕組みが必要になる（カメラをまたぐ再IDなど）が、今回のスコープ外とした。

## Tech-5: 割り当て問題とハンガリアン法

### 毎フレームが「割り当て問題」

カメラは毎フレーム「座標リスト」を返す。これを既存のトラックに割り当てるのが追跡の本質。

```
フレームN+1 の検出結果:   既存トラック:
  検出A (cx=120)            track_0 (最後に cx=118)
  検出B (cx=200)            track_1 (最後に cx=198)
  検出C (cx= 60)            track_2 (最後に cx=155)
```

どの検出をどのトラックに対応させるか？ → これが **割り当て問題**。

---

### 3つの戦略の違い

#### naive（旧実装）
リストを先頭から走査し、30px以内の**最初の**トラックに割り当てる。

```
検出A → track_0 を先に見つけたのでマッチ  （距離2px）
検出B → track_1 にマッチ                   （距離2px）
検出C → track_2 は155px先で圏外 → 新規登録
```

問題: リストの順番次第で、本来マッチすべきでないトラックに割り当てられることがある。

#### greedy（現在の推奨実装）
全ての（検出, トラック）ペアの距離を計算し、**近い順に**割り当てていく。一度割り当てたら再利用しない。

```
全ペアを距離でソート:
  (検出A, track_0): 2px   ← まずここを割り当て
  (検出B, track_1): 2px   ← 次にここ
  (検出C, track_2): 95px  ← 閾値30px超 → 新規登録
```

naiveより正確だが、まだ**局所最適**（近い順に取るだけで全体の最小化ではない）。

#### ハンガリアン法（将来候補）
**全体のコストが最小になる割り当て**を数学的に求めるアルゴリズム。

```
コスト行列（距離）:
           track_0  track_1  track_2
検出A  [    2px      100px    60px  ]
検出B  [  100px        2px    80px  ]
検出C  [   60px       80px    95px  ]

→ 全組み合わせの中から「合計距離が最小」の割り当てを選ぶ
→ A→track_0, B→track_1, C は閾値超えで新規
```

greedy との違いが出るのは、**近いペアが競合するとき**。例えば検出Xと検出Yが両方とも同じtrack_Zに近い場合、greedy は一方を奪い合ってしまうが、ハンガリアン法は全体を見て最適解を出す。

計算量は O(n³)（n = 検出数）。駐輪場規模（n < 30 程度）では十分高速。

---

### まとめ: 3戦略の比較

| 戦略 | 保証 | 計算量 | 実装済み |
|---|---|---|---|
| naive | なし（リスト順依存） | O(n×m) | ✅ |
| greedy | 局所最適（近い順） | O(n×m log nm) | ✅ |
| ハンガリアン法 | 全体最適 | O(n³) | 将来課題 |

## Tech-6: IoUベースのマッチング

### IoU（Intersection over Union）とは

2つのバウンディングボックスが**どれくらい重なっているか**を表す指標。

```
    ┌─────────┐
    │    A    │
    │    ┌────┼────┐
    │    │交差│    │
    └────┼────┘  B │
         └─────────┘

IoU = 交差面積 / (A面積 + B面積 - 交差面積)
```

値は 0〜1。完全に一致すると 1.0、まったく重ならないと 0.0。

物体検出のモデル評価（mAP の計算）でも使われる標準的な指標。

---

### トラッキングへの応用

**座標近傍マッチング（現在）** は重心点の距離だけ見る。

```
フレームN:   自転車の重心 (cx=120, cy=200)
フレームN+1: 自転車の重心 (cx=118, cy=203)  → 距離 2.8px < 30px → 同一個体
```

**IoUベースのマッチング** はバウンディングボックス全体の重なりを見る。

```
フレームN:   BB = (X=80, Y=160, x=160, y=240)
フレームN+1: BB = (X=78, Y=158, x=158, y=238)  → IoU ≈ 0.95 → 同一個体
```

---

### 密集時に強い理由

自転車が隣り合って駐まっているとき、重心の距離だけでは区別が難しい。

```
自転車A: 重心 cx=100
自転車B: 重心 cx=130
→ 30px ちょうど。どちらにマッチすべきか不定になりやすい
```

IoUを使うと、**ボックスの形・サイズ・位置を総合的に判断**できるため、重心が近くても別個体と判別しやすい。

---

### 実装コスト

座標近傍マッチングと比べて計算は重くなるが、フレームレートが低い（現在1fps程度）今回の用途では問題にならない。

課題3（密集時の誤マッチ）の根本解決として有力な将来候補。

# 現状の課題：個体識別ロジック

## そもそも「個体識別」とは何をしているか

AITRIOSカメラは、フレームごとに「この場所に自転車がある」という**バウンディングボックス（BB）**を出力するだけで、
「この自転車は前のフレームの何番と同じ個体か」という**ID情報を持たない**。

そこで今回は、**座標の近さで同一個体を推定する**という独自ロジックを実装した。

```
フレーム1: (cx=120, cy=200) に自転車を検出
                 ↓ 次のフレームで 30px 以内に検出 → 同一個体とみなす
フレーム2: (cx=118, cy=203) に自転車を検出  → track_id=0 として滞在時間を計算
```

このロジックを **座標近傍マッチング（Coordinate Proximity Matching）** と呼ぶ。

---

## 課題1：最近傍が保証されない

### 何が起きているか

現在の実装は「30px以内の最初のトラック」にマッチする。
検出が複数あるとき、距離が一番近いトラックではなく、**リストの先頭のトラック**に当たる。

```
トラックA: cx=100  ←  距離12px
トラックB: cx=108  ←  距離4px（本当はこちらが正解）

現在: トラックA が先にリストにあれば → トラックAにマッチ（誤り）
```

### 理想

全検出 × 全トラックの距離を計算し、**コスト最小の組み合わせ**で割り当てる（ハンガリアン法）。

---

## 課題2：消えた自転車のトラックが残り続ける

### 何が起きているか

一度登録されたトラックは **永遠に消えない**。

駐輪場から自転車が去っても、そのトラックがリストに残り続けるため、
後から来た別の自転車が「同じ個体」と誤認識される可能性がある。

```
09:00  自転車Aが来る   → track_id=0 として登録
09:30  自転車Aが去る   → track_id=0 はリストに残ったまま
10:00  自転車Bが来る   → track_id=0 の近くなら「自転車A継続」と誤認識
                         （本当は新しい自転車B）
```

### 理想

`last_seen` から一定フレーム（例: 5フレーム）以上検出されなければ
トラックを**消滅（Dead）** させる。

---

## 課題3：密集時の誤マッチ

### 何が起きているか

自転車が隣り合って止まっていると、30pxの半径が重なり、
どちらのトラックにマッチするか不定になる。

```
  [トラックA]  [トラックB]
   cx=100        cx=125
       ← 25px →
         ↑ 半径30pxが重なる
```

新しく検出した自転車が本来B側なのにAにマッチしてしまうと、
AとBの滞在時間が両方ずれていく。

### 理想

マッチング半径をBBのサイズに応じて動的に変えるか、
BB同士の **IoU（重なり率）** でマッチングする。

---

## まとめ

| # | 課題 | 影響 | 改善アイデア |
|---|---|---|---|
| 1 | 最近傍が保証されない | 誤トラック割り当て | ハンガリアン法で最小コスト割り当て |
| 2 | トラックが消えない | 別個体を継続と誤認識 | 一定フレーム未検出でトラック消滅 |
| 3 | 密集時の誤マッチ | 滞在時間の誤算 | IoUベースのマッチング |

現時点では課題1・2が発生しやすく、**滞在時間の計算精度に直結する**ため優先度が高い。
課題3は駐輪場の密度が高い場合に問題になるが、今回のデータでは限定的。
